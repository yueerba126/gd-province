### 模块说明

| 模块      | 含义   | 说明                                                             |
| --------- |------|----------------------------------------------------------------|
| home      | 主模块  | 聚合各个模块统一打包成一个服务                                                |
| framework | 基础框架 | 缓存、安全控制(签名摘要、数据加密)、数据绑定、文件存储、持久层、限流                            |
| organize  | 组织架构 | 行政区域、行政组织、企业组织、企业信用、人员、角色、菜单、功能权限                              |
| basis     | 基础信息 | 库点、仓库、廒间、货位、油罐、设备、药剂、文件、仓房点位<br/>视频设备点位                        |
| trade     | 粮食购销 | 合同、入库、入库检验、入库结算、出库、出库结算<br/>倒仓、粮食库存、损益单、粮食性质转变、账面库存、客户         |
| manage    | 粮库管理 | 安全管理、温湿度检测、害虫检测、气体检测、通风作业<br/>熏蒸作业、仓内视频图像、视频监控异常事件告警、违规预警、质检数据 |
| admin     | 行政管理 | 储备规模、储备计划、轮换计划、项目信息                                            |
| common    | 公共模块 | 存放公共信息                                                         |
| collect    | 数据收集 | 用于对接各个企业的数据                                                    |
| reserve    | 储备计划管理 | 储备计划管理、储备规模管理

### 规范说明

- 后端请求都为POST请求
- 接口路径必须全小写，驼峰单词使用/分割
- 接口必须校验参数合法性，不能只靠前端校验
- 管理端接口必须校验用户权限，前端只能防护普通用户
- 运用缓存降低数据库查询压力，缓存一些不经常变动的数据

- SQL查询条件必须命中索引

- SQL不允许超过三张表关联查询

### 性能指标

- 吞吐量：TPS=1000 QPS=1000
- 后端：对接API接口最大响应时间100ms，管理系统接口最大响应时间300ms
- 前端：首页反馈时间最长1000ms，子页面反馈时间最长100ms，保证页面操作流畅度

### 分支管理

| 分支    | 作用 | 说明                                          |
| ------- | ---- |---------------------------------------------|
| dev     | 开发 | 基于最新开发的代码                                   |
| test    | 测试 | 基于开发环境自测通过的代码                               |
| release | 封版 | 基于测试环境测试通过的代码,格式为[release-日期]用于记录每一次发版的日期序号 |
| master  | 发布 | 基于最新的release预发布分支                           |

**分支合并流程**

- dev开发完成以后在开发环境自测通过后合并到test分支
- 测试环境测试通过以后确定可以封版,基于test分支 构建[release-日期] 预发布分支
- 将本次预发布分支合并到master主分支 上线打包基于master主分支的代码进行发版

**发布流程**

- 发布前会先执行db/migration 下面的SQL脚本 -- 人工手动执行《线上环境会关闭自动执行》
     例如本次预发布分支是release-2023-05-16，会根据上一次的release分支确定要执行多少个脚本
     比如说上次发布的是release-2023-05-01 那么对应本次发布执行的脚本是2023-05-02到2023-05-16
- 将master主分支的代码进行打包发版

**紧急修复**

- 如果说release-2023-05-16的代码有问题，需要紧急修复要基于release-2023-05-16之上进行修复,修复完以后再合并到master再发布

**延期修复**

- 直接在dev上修复，等到下一个版本发布时将新功能和修复BUG一起发布

**版本回退**

- 出现了重大问题，修复不了也运行不了.打包上一个预发布分支的代码重新发布,执行sql回退脚本,撤销本次发布带来的影响
